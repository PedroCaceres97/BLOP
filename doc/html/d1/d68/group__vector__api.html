Main operations for creation, insertion, deletion, and traversal.

Warning
Thread Safety
None of the functions are thread-safe. Caller must provide synchronization when using from multiple threads.

void blopfn_vector_clear ( blopt_vector vec)

Remove all elements from the vector.

Complexity:

  • O(n) if BLOP_VECTOR_DEALLOCATE_DATA is defined (deallocation per element)
  • O(1) otherwise

Ownership rules:

  • If BLOP_VECTOR_DEALLOCATE_DATA is defined, the macro is called for each element before freeing storage.

◆ blopfn_vector_create()

blopt_vector blopfn_vector_create ( )

Create and initialize a new empty vector.

Ownership:
Caller owns the returned vector and must destroy it with blopfn_vector_destroy.

Complexity: O(1)

Returns
Valid vector handle, or NULL on allocation failure.
Remarks
The returned vector has capacity BLOP_VECTOR_INITIAL_COUNT and count == 0.

◆ blopfn_vector_destroy()

void blopfn_vector_destroy ( blopt_vector vec)

Destroy a vector previously created with blopfn_vector_create.

Complexity: O(1)

Parameters
vecVector to destroy.
Warning
Precondition: The list must be empty (size == 0).
Ownership: Frees the vector structure itself and vector->data. Does not free individual data.
Remarks
Use blopfn_vector_clear to remove elements first (and optionally deallocate elements via BLOP_VECTOR_DEALLOCATE_DATA).

◆ blopfn_vector_insert()

void blopfn_vector_insert ( blopt_vector vec,
size_t idx,
BLOP_VECTOR_DATA_TYPE value )

Insert value at position idx, shifting subsequent elements to the right.

Real time complexity is n - idx if no resizing is needed Complexity: O(n)

Parameters
vecVector to modify.
idxInsertion index (0..count). Inserting at count is equivalent to push_back.
Remarks
It is valid to insert at count index, altough it may look like out of bounds it perfectly makes if you consider push_back() where no subsequent elements gets shifted to the right
Note
The implementation optimizes boundary cases (push_front / push_back).

void blopfn_vector_shrink ( blopt_vector vec)

Internal helper that reduces capacity if vector is significantly under-utilized.

Complexity: O(n)

Parameters
vecVector to examine/shrink.
Note
The shrink policy in the header reduces capacity when: count < capacity / (BLOP_VECTOR_SCALATOR * 2) && count > BLOP_VECTOR_INITIAL_COUNT.
