Memory layout and pointer arithmetic.

Memory layout and pointer arithmetic.

  • Each allocation is represented by a struct blops_ptrhdr placed immediately before the payload returned to the caller.
  • The public pointer is computed as (void*)((char*)hdr + sizeof(blops_ptrhdr)).
  • Header retrieval is performed with the inverse subtraction.
Remarks
Caller code must not assume any particular alignment beyond that provided by the platform malloc. The header layout relies on standard C memory layout rules.

Ownership & validation

  • Allocations created with a pool function are owned by that pool until blopfn_pool_free is called.
  • blopfn_pool_free validates that the pointer being freed belongs to the provided pool. Passing a pointer not allocated by the pool triggers an assertion/error.
  • blopfn_pool_realloc and blopfn_pool_duplicate require the source pointer to be tracked by the same pool.

Resource accounting

  • pool->bytes is increased by the size argument on each allocation call and decreased when allocations are freed.
  • pool->ptrs.size reflects the number of outstanding allocations.

These invariants are maintained even across realloc/duplicate operations.

  • Allocation failures call BLOP_ERROR_MESSAGE and BLOP_ABORT (project-wide policy defined in blop.h).
  • Passing a NULL pool or NULL pointer to functions that require non-NULL arguments will trigger BLOP_ASSERT macros (behavior depends on compile-time assert configuration).
  • Attempting to free or operate on a pointer that does not belong to the provided pool is considered a fatal error (assert/abort).
